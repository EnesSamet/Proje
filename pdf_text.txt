Project Speciﬁcations 
Project Objective 
The aim of this project is to design and implement a fully functional GUI -based Java-
based automation system  that manages a real -world organizational process in a 
structured, reliable, and user -friendly way. The system should automate routine 
operations such as storing records, updating information, processing transactions, 
generating outputs, and controlling user interactions. 
You are expected to apply core software engineering principles throughout the project, 
including requirements analysis, system design, implementation, testing, 
documentation, and presentation. 
 
General Project Requirements 
Each project must: 
• be developed in Java  
• solve a realistic automation problem  
• include record management operations  
• support user interaction  
• process and store data systematically  
• include validation and error handling  
• be modular and maintainable  
• be documented properly  
• be testable and demonstrable  
Minimum Functional Requirements 
The system must include at least the following features: 
• User Interface with multiple screens/panels 
• User login or role-based access control if appropriate for the project  
• Add new records  
• Update existing records  
• Delete records  
• Search and ﬁlter records  
• Display records in tables or lists (e.g., JTable)  
• Process at least one transaction-based operation  
• Generate summary information or reports  
• Input validation  
• Exception handling  
• Persistent data storage (ﬁle or database) 
• Feedback messages (success/error notiﬁcations) 
Technical Requirements 
Students must use Java and are expected to apply appropriate object -oriented 
programming principles. 
The project should include: 
• classes and objects  
• encapsulation  
• inheritance where appropriate  
• polymorphism where appropriate  
• ﬁle operations or database usage for data persistence  
• clear menu structure or interface design  
• layered and modular code structure  
GUI Requirements 
The project must be a GUI-based application. 
The interface must: 
• The system must include a user-friendly graphical interface  
• be intuitive and user-friendly  
• include buttons, labels, text ﬁelds, tables, and menus  
• support navigation between screens  
• display structured data clearly  
• provide visual feedback to the user 
• Console-based applications are NOT allowed  
• Acceptable technologies:  
o Java Swing  
o JavaFX  
Projects that do not include a GUI will NOT be evaluated. 
Data Storage 
For data storage, one of the following may be used: 
• text ﬁles  
• CSV ﬁles  
• JSON ﬁles  
• SQLite / MySQL or another suitable database  
Required System Components 
The software should contain the following components: 
1. User Interface Layer 
This is the part where users interact with the system. It should be clear, consistent, and 
easy to use. 
2. Business Logic Layer 
This layer should implement the automation rules, workﬂows, decisions, and 
operational logic of the system. 
3. Data Management Layer 
This part should manage reading, writing, storing, updating, and retrieving data. 
4. Reporting or Output Layer 
The system should produce meaningful outputs such as lists, summaries, statistics, or 
transaction results. 
Software Engineering Deliverables 
Each group must submit the following: 
A. Requirements Document 
This document must clearly deﬁne: 
• project purpose  
• target users  
• system scope  
• functional requirements  
• non-functional requirements  
• assumptions and constraints  
B. Design Document 
This document must include: 
• system architecture  
• class design  
• data design  
• interface design  
• UML diagrams  
At minimum, students should prepare: 
• Use Case Diagram  
• Class Diagram  
• Flowchart or Activity Diagram  
C. Source Code 
The code must be: 
• readable  
• well-structured Java code 
• commented where necessary  
• Proper naming conventions 
• logically organized into packages or modules  
D. Test Document 
Students must test the system and document: 
• test scenarios  
• test cases  
• expected results  
• actual results  
• detected errors and ﬁxes  
E. Final Report 
The ﬁnal report should combine and explain the entire project development process, 
including analysis, design, implementation, testing, conclusions, and future 
improvements. 
F. Presentation 
Each group must present the project in class and demonstrate the system live. 
Rules 
Quality Rules 
• Hard-coded data only is not sufficient unless clearly justiﬁed.  
• Poorly structured code with all logic inside one class will not be accepted.  
• Copy-paste level interfaces without system logic will not be considered 
sufficient.  
• Superﬁcial projects with minimal functionality will receive lower grades. 
Teamwork Rules 
• Each group member should contribute to analysis, coding, and reporting.  
• During the presentation, all members should be able to explain the system.  
• Individual contribution may be questioned during evaluation. 
Submission Requirements 
Each group must submit: 
1. Project report in PDF

4. UML and design documents

Submission & Development Policy  
GitHub Usage 
Students are allowed to use GitHub only under the following conditions: 
• All repositories must be PRIVATE during the entire semester  
• Public repositories are strictly forbidden before grading is completed  
• Students must not share their repositories with anyone outside their group  
Plagiarism Policy  
Plagiarism is taken very seriously in this project. 
The following actions are strictly prohibited: 
• Copying code from other students  
• Sharing code between groups  
• Using publicly available project solutions without proper modiﬁcation  
• Uploading code to public platforms during the semester  
All submissions will be checked for: 
• Code similarity  
• Structural similarity  
• Development patterns  
If plagiarism is detected: 
• The project may receive a zero grade  
• Additional disciplinary action may be applied 
Development Process Requirement (Commit Tracking) 
Students are expected to follow a progressive development process, not a one-time 
submission. 
Therefore: 
• The project must include a version history  
• Students should make regular commits during development  
• Each commit must include meaningful comments/messages  
 Example of good commit messages: 
• "Implemented GUI form for record entry"  
• "Added validation for input ﬁelds"  
• "Fixed bug in update functionality"  
• "Integrated database connection"  
Unacceptable commit messages: 
• "update"  
• "ﬁnal"  
• "ﬁx"
